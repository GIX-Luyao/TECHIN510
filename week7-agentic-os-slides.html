04 / 24
The Problem

Why a Single Agent Is Not Enough

Plain Claude / one-shot agent
  • Has no idea who you are
  • Forgets every session
  • Asks the same five questions each time
  • Cannot run on a schedule
  • Cannot route work to the right specialist

"Help me write a cover letter." → Generic. Hedged. No memory of yesterday's draft.

Agentic OS
  • Loads identity + user profile on every session
  • Appends to a daily memory file you can read
  • Carries open threads forward
  • Cron-style scheduled jobs
  • Skills auto-route by trigger phrase

"Help me write a cover letter." → Pulls your resume, prior research, last week's draft, and your voice rules.

06 / 24
Before You Write a Skill

Three Brand Layers, In Order

1
Voice profile — how it speaks Tone, banned words, defaults under pressure. Lives in SOUL.md.
2
Positioning — what it is and is not The one-line answer to "what does this agent do?" + an explicit non-goals list.
3
Ideal customer profile — who it is for One real persona, not a market segment. Skills tune output to this person.

Skip these and every skill you write will sound like a generic chatbot wearing a costume.

Example — Job Application Agent

Voice Profile in SOUL.md

# SOUL.md — Agent Personality

You are a personal job-application agent. Think of yourself as a sharp
friend who has read every "how to land an internship" guide.

## Voice
- Direct. Lead with the answer. No throat-clearing.
- Honest about gaps. If the user lacks a qualification, say so.
- Concrete over abstract. Name the bullet, name the company.
- Calm, not hyped. No emoji. No "absolutely crushing it."

## Words to avoid
leverage, synergy, robust, passionate, journey, in today's world

Voice is the most undervalued line of defense against AI slop. Two paragraphs in SOUL.md beat 1,000 lines of prompt engineering downstream.

  • 3rd-year UW Informatics student
  • Targeting agentic-AI internships, fall back to FAANG
  • Has 4 named projects with metrics
  • Two named weaknesses to address honestly
  • Dislikes the words "leverage", "synergy", "robust"

A real ICP — not a market segment. The agent tunes everything to this student.

The Four Context Files

Each File Owns One Thing

File Owns Lifecycle Who edits
SOUL.md Personality, voice, principles, non-goals Rarely changes Human
USER.md Who the user is, strengths, weaknesses, preferences Slow drift Human (agent suggests)
memory/{YYYY-MM-DD}.md Session #, project, goal, deliverable, open threads One file per day Agent appends
learnings.md What worked / didn't / lessons per skill, after shipping each deliverable Append-only Agent (human reviews)

One concern per file. If two files want to own the same fact, the design is wrong.

11 / 24
Identity Layer

SOUL.md vs. USER.md

SOUL.md — about the agent

Bootstrap prompt:

Ask me 5 questions so we can agree
on your personality: what I expect,
what I disallow, working principles.

Voice, principles, non-goals. Stable across users of the same template.

USER.md — about the human

Bootstrap prompt:

Ask me 15 questions one at a time
to update @context/USER.md:
identity, likes/dislikes, what to
avoid, preferences, goals &
working style.

One per person. Forks when you onboard a new user.

Both load on every session via @context/SOUL.md and @context/USER.md imports in CLAUDE.md.

## Session 1

Apply to Anthropic SWE internship by Friday.

### Deliverable
- projects/job-tailor-resume/2026-05-09_anthropic-swe.md
- projects/job-cover-letter/2026-05-09_anthropic-swe.md

### Open threads
- CS 452 enrollment confirmation — mention in follow-up email if asked.
- Distributed-systems gap — address honestly in next draft.

Five fields per session: session #, project, goal, deliverable, open threads. New sessions read today's file and yesterday's — open threads become tomorrow's starting context.

13 / 24
Compounding Layer

learnings.md — What Worked, Per Skill

When it's updated
  • After shipping each major deliverable
  • Per skill: what went well
  • Per skill: what did not go well
  • Per skill: what should be learned next time

Wired into CLAUDE.md so the agent updates it automatically.

What it does NOT store
  • One-off session details (those go in memory)
  • Identity facts (those go in USER.md)
  • Skill mechanics (those go in the skill itself)
  • Anything the agent isn't sure about
# context/learnings.md — append-only

## draft-cover-letter — 2026-05-08
- Went well: leading with the project, not the role.
- Didn't: opening with "I am writing to express." User rejected x3. Banned.
- Learn: pull the hook from the user's most recent shipped artifact.

## draft-resume — 2026-05-09
- Went well: metrics in 2nd clause — "Built X, cutting Y by Z%".
- Didn't: padding bullets to fill space.
- Learn: cut to 4 bullets per role unless the user asks otherwise.

16 / 24
The Boundary

What a Skill May Update On Its Own

Surface Auto-update? Why
memory/{date}.md Yes — append Append-only ledger; agent owns it
learnings.md Yes — append after each shipped deliverable Per-skill: went well / didn't / learn
job_context/companies.md Yes — append a paragraph Owned by job-research-company per the matrix
applications.md Yes — append, never overwrite Audit trail must not be rewritten
USER.md Suggest only Identity is human-edited; agent proposes a diff
SOUL.md No Personality is a contract. Humans only.
SKILL.md body No A skill rewriting itself is how silent drift starts
17 / 24
Update Protocol

When Skills Update Themselves

1
After every deliverable → append to today's memory Goal, deliverables, decisions, open threads. Silent. No "I logged that."
2
After shipping each major deliverable → append to learnings.md What went well, what didn't, what to learn next — per skill. Wired in CLAUDE.md.
3
End of every session → reconcile Update memory, log learnings, propose personality tweaks if needed.
4
Human-only updates surface as a diff, never as silent edits USER.md / SOUL.md changes: "Want me to add 'Go (learning)' to your skills line?"

Iron rule: the agent's writes are auditable. Open the file, read what was added. If you can't tell what changed, the write was wrong.

The Skill Contract

Every Skill Declares Its Relationships & I/O

# .claude/skills/draft-cover-letter/SKILL.md

## Upstream (depends on)
- draft-resume          → tailored resume bullets
- research-company      → company brief + angle

## Downstream (feeds)
- track-application     → logs the submitted artifact

## Input      job posting URL, target company name
## Output     markdown cover letter, ~350 words
## Format     {date}_{company}_{role}.md
## Stored at projects/job-cover-letter/
Each skill must specify
  • Upstream + downstream skills
  • What it takes as input
  • What it produces as output
  • Format + where to store
Why this matters
  • Skills compose without calling each other
  • Every handoff is a file you can cat
  • Add or remove a skill without breaking the chain

Self-Audit

Skill & MCP Reconciliation

# Excerpt from CLAUDE.md

## Skill reconciliation
On session start, compare what is on disk in .claude/skills/
against what is registered in this file and learnings.md.

- Additions: register silently. Add to the skill table below.
- Removals: confirm with the user before deleting entries.
- Drift: if a skill's input/output spec on disk diverges from
  the registered version, surface a diff — never edit silently.
The asymmetry
  • Adding a capability is safe → auto-register
  • Removing one breaks chains → require confirmation
  • Same rule applies to MCP servers
Why this is in CLAUDE.md

It runs every session. The OS audits itself before it does anything else — the same way fsck runs before mount.

The Cron Daemon

Scheduled Jobs — crons/jobs/

Folder shape
job-agent/
├── crons/
│   └── jobs/
│       ├── daily-job-scan.md
│       ├── weekly-applications-digest.md
│       └── monthly-resume-refresh.md
└── scripts/
    └── start-crons.sh   # registers all jobs

Jobs are markdown files. The shell script reads them and schedules system cron entries.

A job file
# crons/jobs/daily-job-scan.md
schedule: 0 8 * * *
skill: research-company
input: watchlist.md
output: companies.md (append)

## What it does
Pulls fresh openings for tracked
companies and appends new roles
to the brief. Surfaces matches.

No external service required. The whole agentic OS runs from local files + system cron.
