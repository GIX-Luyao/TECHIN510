Real-World Breach

A Recent Real-World Breach

OpenReview / ICLR 2026 — November 2025

Around 10,000 papers’ worth of reviewer, author, and area-chair (senior reviewer) identities were de-anonymized and circulated publicly. Once names became public, third parties harassed and even tried to bribe reviewers. OpenReview patched the bug within an hour.

The cause: an API endpoint returned full user profiles for any conference on the platform with no check that the caller was allowed to see them. OWASP calls this class of bug Broken Access Control. Logging in worked. The authorization check did not exist.

A working app moves data across boundaries. A safe app validates each crossing. This week we learn the three checks that would have stopped this breach.

The Distinction

Authenticated ≠ Authorized

Authentication

Definition: the process of verifying that a user is who they claim to be — usually by checking credentials.

After signing in, the user holds a valid session token. The system can now recognize this person across requests.

Authorization

Definition: the process of deciding what an authenticated user may access or change.

For each piece of data, the system asks: is this specific user allowed to read or write this row?

The demo had authentication.
It had no real authorization.

Pillar 1 — Auth Establishes Who

Sign-Up vs. Sign-In

Sign-Up — one time
1
Collect identity Email, password, optional confirmation.
2
Create the account record Supabase writes a row in auth.users.

API: supabase.auth.signUp({ email, password })

Sign-In — every session
1
Verify the credentials Prove the person is the account owner.
2
Mint a session token Browser receives a JWT for this session.

API: supabase.auth.signInWithPassword({...})

Memory rule: Sign-up creates identity. Sign-in creates access.

07 / 24
Where Passwords Must Not Live

Why Passwords Live in the Auth Server

A password is the most sensitive secret a user gives you. Where you put it and how you store it decide whether a single bug, leaked backup, or stray log line turns into a breach. The industry has learned this the hard way.

Wrong places — what went wrong
A
In the browser (React state, localStorage) Any XSS, browser extension, or shared device reads it in plaintext. British Airways 2018: injected script skimmed credentials from 380K users in the page itself.
B
In your own database, weakly hashed LinkedIn 2012: 167M unsalted SHA-1 hashes leaked — most cracked within days. Adobe 2013: 153M reversibly-encrypted passwords + plaintext hints exposed.
C
In application logs or plaintext files Facebook 2019: 200M–600M passwords found stored in plaintext in internal logs, searchable by ~20K employees over years. RockYou 2009: 32M plaintext passwords still seed credential-stuffing attacks today.
Right place — the auth server
1
Hashed with bcrypt/argon2, per-user salt A leaked row gives an attacker a slow, costly hash — not a password.
2
Isolated from your app code & tables Your queries, logs, and frontend never touch the raw password — so they cannot leak it.
3
Your app holds a JWT, not the password { sub: user_id, exp: timestamp } — a short-lived signed claim. The server verifies the signature on each request.

Rule of thumb: if your code can read the password, your breach can leak it. Let Supabase Auth hold the secret; your app holds the JWT.

The Bridge to Ownership

The User ID Is the Anchor

After sign-in, the server can read

const { data: { user } } = await supabase.auth.getUser();
const userId = user.id;
// e.g. "8f3c1a2e-4e72-4...-9b6d"

UUID (Universally Unique Identifier): a 36-character random string that identifies one user globally and never collides with another.

This UUID is the anchor. It comes from the verified session. The same value is stored on every row this user owns, and the same value appears inside every RLS policy as auth.uid().

User_id at three places: session → row → policy.

Pillar 1 — Guarding the Door

Protected Routes

Recipe: 3-step guard order
1
Loading state first Show neutral UI while identity is unknown.
2
Server-side session check Verify on the server (middleware or server component) before any private content renders.
3
Only then render content Authenticated users see the page; everyone else is redirected.

Order matters. Render-then-check leaks data.

Pillar 2 — Ownership Tags What

Screen → Spec → Schema → RLS

Every screen rule should land in a database rule. If the dashboard says “My Tasks,” the schema needs a user_id column, and the RLS policy must check it. A traceable line from UX intent to database enforcement.

Privacy is a design decision before it is a database column.

11 / 24
All Four Operations

CRUD With Ownership

CRUD = Create, Read, Update, Delete — the four basic operations on stored data.

OperationOwnership ruleFailure mode if you skip it
Create Attach user_id from the session Row has no owner, or worse: a forged owner
Read Scope queries to the current owner Every authenticated user reads every row
Update Reject if row owner ≠ current user Anyone can edit anyone’s data
Delete Reject if row owner ≠ current user Anyone can delete anyone’s data

AI often generates code that works before it generates code that enforces ownership.

Pillar 3 — RLS Enforces the Boundary

Why App Logic Is Not Enough

Equifax — September 2017 (disclosed)

About 147 million Americans — plus ~15M in the UK and ~19K in Canada — had records exposed: Social Security numbers, dates of birth, home addresses, driver’s license numbers, roughly 209,000 credit-card numbers, and ~182,000 dispute documents containing further PII. Settlement: up to $700M with the FTC, CFPB, and state AGs.

What a second layer would have done

Each layer trusted the one before it. The web tier trusted the patch, the app trusted the web tier, the databases trusted the app’s service account — so one missed update drained everything.

If the database itself had enforced row rules — auth.uid() = user_id — a stolen connection would inherit one user’s scope, not 147 million. The patch failure becomes a contained incident, not a national one.

One missed patch should not expose everything. Defense in depth: app-level checks and RLS — not one or the other.

Defense in Depth

Every Boundary Needs a Rule

BoundaryRule toolWhat it catches
Browser → Server Form validation + Zod schema Bad input blocked before processing
Server fn → Server fn TypeScript interface + auth check Wrong types and unauthenticated callers
Server → Database SQL constraints + parameterized queries Impossible rows, injection attempts
Database → User auth.uid() RLS policy Cross-user reads and writes

Code-level filters and database-level enforcement are two different lines of defense, not duplicates of each other.

14 / 24
RLS — Step 1

Enable → Deny All

Row-Level Security (RLS): a database-level rule that decides — per row, per query — which rows each user is allowed to read or change. It runs inside Postgres, not inside your app.

Two SQL lines, big consequence

alter table tasks enable row level security;

The instant that runs, every app-client select against tasks returns 0 rows. The app appears broken.

Deny-by-default is not a bug.
It is the starting point.

Mental model: RLS is a bouncer who says “no” until you write a policy that lets specific people in. Supabase dashboard and service-role access can bypass this; your browser app should not.

15 / 24
The Distinction Most People Miss

Write Policies: using vs. with check

OperationClauseWhat it asks
SELECT using Which existing rows may this user see?
DELETE using Which existing rows may this user remove?
INSERT with check Is the new row state allowed?
UPDATE both May this user touch this row, AND is the new state allowed?

Common bug: you write a using policy, feel safe, and forget with check. Inserts can smuggle bad rows; updates can change ownership unless the new row state is checked.

16 / 24
Same Query, Two Outcomes

Without RLS vs. With RLS

Without RLS

select * from tasks;

12 rows

Alice sees Alice’s rows, Bob’s rows, and everyone else’s. The query is identical.

With RLS

select * from tasks; -- using ( auth.uid() = user_id )

3 rows

Postgres silently appends the policy to every read. Alice sees only Alice’s tasks. App code unchanged.

The chain’s last link is visible as a row count.

When the Chain Breaks

Real-World Damage: Four Failure Patterns

Broken object access

First American (2019) — predictable URLs, no ownership check.

W6 habit: check ownership on every read, every URL.

Weak internal boundaries

Equifax (2017) — one missed patch, no second line of defense.

W6 habit: layer defenses — app checks and RLS.

Leaked credentials

Uber (2022) — admin credentials in a script, plus MFA (Multi-Factor Authentication) push fatigue.

W6 habit: protect keys — service-role keys never in client code.

Student data exposure

Chegg (multiple) — weak controls on data the company didn’t need to keep.

W6 habit: minimize sensitive data; collect only what the JTBD requires.

Four very different breaches. Four habits you build tonight.

If any expected cell turns out non-zero, your chain is broken.

Validation Is a Boundary

Every form is a door into your database
1
Required fieldsReject blank values before they become impossible rows.
2
Length limitsStop a 10,000-character paste from becoming a UI and database problem.
3
Type checksEmail should look like email; status should be one of the allowed choices.
