The Security Demo

Every Row, Every User

“Every task in this database — every to-do, every note, every piece of personal data — is visible to any authenticated user. Not just their own tasks. Everyone’s.”

If this were a medical records system, that’s a HIPAA violation. If it were a university system, that’s FERPA. If it were a financial app, that’s potential litigation.

This isn’t a bug in the code. The code works perfectly. This is an architectural failure.

Architecture

Three-Layer Architecture

Three separate locations, three different security boundaries. Defense in depth: every layer checks.

Defense in Depth. Client checks catch mistakes. Server checks enforce rules. RLS enforces ownership. No single layer is trusted alone.

Definition: Defense in depth means multiple independent checks protect the same data.

Why it matters: if one check fails, another check can still prevent a breach.

Example: university grade data is protected by page access rules, API checks, and row-level ownership in database.

Design Principle

Screen → Spec → Schema → RLS

“The screen is the spec. The spec drives the schema. The schema drives the RLS policy.”

If the dashboard shows “My Tasks,” then the tasks table must have a user_id column, and the RLS policy must filter on that column. A direct, traceable line from UX decision to database security rule.

AI principle: AI writes code layer by layer. If you don’t understand the layers, you can’t verify what the AI built — or catch the RLS it forgot to enable.

Protected Routes

Who is this user, and what can they access?

By the end of this section, you can explain this full chain: sign-up -> sign-in -> session -> route guard -> user_id -> RLS.

Authentication: confirm who the user is. Authorization: decide what that user can access.

Protected route: a page only authorized users can open.

Example: all students can log in, but only advisors can open advising admin pages.

10 / 37
Authentication

Sign-Up vs. Sign-In (Plain Language)

Sign-Up = make a new account (one time)
1
Collect identity information Email, password, and account confirmation method.
2
Create account record System stores the new user as a long-term identity.

Real-world scenario: Student A creates a campus planner account before the quarter starts.

Sign-In = start a session (every login)
1
Verify account owner System checks credentials and confirms this person is the account owner.
2
Start session System issues temporary proof of identity for this browsing session.

Real-world scenario: Student A returns tomorrow, signs in, and regains access to only their own tasks.

Quick memory rule: Sign-up creates identity. Sign-in creates access.

Class prompt: In a food-delivery app, what problems appear if you only support sign-up but not sign-in sessions?

11 / 37
Authentication

What Is a Session?

“An account identifies you once. A session proves you are still logged in right now.”

A session is the logged-in state. Your app checks it on each protected page load.

What is stored?

A JWT (JSON Web Token): a signed token that says who the user is for this session.

What is NOT stored?

The user's raw password. Password checking stays inside Supabase Auth.

Why it matters: without a valid session, app cannot safely tell if the visitor should access private data.

Example: online banking asks you to log in again after inactivity because old sessions can expire.

Transition: once session exists, next question is "How do we protect a route before content appears?"

Protected Routes

Common Bug: Flash of Protected Content

Failure pattern (no code needed)
1
User opens protected page Dashboard starts to load before auth status is confirmed.
2
Auth check finishes late System discovers user is logged out and triggers redirect.
3
Brief data exposure Content flashes for a moment, which is still a trust violation.
Example walkthrough (Context -> Action -> Outcome)

Context: Student B tries to open Student A's advising dashboard URL.

Action: dashboard begins rendering before the access check finishes.

Outcome: private note flashes briefly, then redirect happens.

Flash-of-content: users see protected content even for 0.2 seconds. That is still a security/trust failure.

Example: a student briefly sees another student's private advising note before redirect.

13 / 37
Protected Routes

The Fix: 3-Step Guard Order

Safe guard sequence
1
Show neutral loading state first Do not show private content while identity is unknown.
2
Verify session status Check whether this user is authenticated right now.
3
Only then choose route Authenticated users see content; others are redirected.

Beginner checklist: loading first -> auth check second -> content last.

Server-side check: access decision is enforced in backend, not only in browser display logic.

Why it matters: client-side redirects can be bypassed; server checks cannot be skipped from UI tricks.

This order is a security rule, not just coding style.

CRUD

Insert Safely: Always Attach user_id

Insert design factors
1
Check login status first No authenticated user means no write operation.
2
Validate input before saving Reject empty, oversized, or malformed values.
3
Attach ownership metadata Every task must include who owns it.

Definition: user_id is the ownership field that marks who owns each row.

Why this matters: if row has no owner, RLS cannot decide who should see it.

Real-world scenario: in a shared apartment app, chores without owner tags become visible/editable by everyone.

Ownership rule: user_id connects a task to one user, then RLS can enforce access.

Data Integrity

Zod: Write Rules Once, Reuse Everywhere

Step 1: Define Input Rules

A
Required fields What must always be present?
B
Allowed format What shape, type, and length are acceptable?

Step 2: Enforce Before Write

A
Reject bad input clearly Return specific feedback so users can recover.
B
Store only validated data Never write untrusted values directly to database.
Input validation (definition): rules that block malformed or unsafe input before it is stored or processed.

Why it matters: invalid input can break data quality and create security risk. Example: airline booking forms check date format in UI, then re-check on server before ticket creation.

Architecture

Every Boundary Needs a Rule

Data quality and security rules at each boundary.
BoundaryRule ToolPlain-Language Effect
Browser → Server Zod schema Bad form input is blocked before processing
Server function → Server function TypeScript interface Wrong data types are caught early
Server → Database SQL constraints Database rejects impossible rows
Database → User auth.uid() (current user ID) RLS Each user only sees their own rows

Data contract (definition): an explicit agreement about what data shape and behavior are allowed at a boundary.

Why it matters: without a contract, systems silently disagree and produce fragile bugs.

Example: if mobile app sends "done" but API expects true/false, task status updates fail unpredictably.

UX Principle

Context Engineering for Users

“Claude streams tokens as they generate. This is not just a performance trick — this is a design decision about trust. While that text appears, you know the system is working.”

We talk about context engineering for giving AI the right information. The same concept applies in reverse: giving users the right information about what the system is doing at any given moment.

Perplexity: “Searching 14 sources…” — that sentence tells you the system is not frozen, and the answer is based on something. Compare that to a blank white screen for 3 seconds.

UX Patterns

Three Patterns That Matter

1
Loading states are context signals A spinner or skeleton screen is the system telling the user “I am working, not broken.” In Next.js, the loading.tsx file does exactly this.
2
Progressive disclosure in AI apps Show results as they become available. Users can begin processing partial information. Perplexity’s source count, Copilot’s grayed-out suggestions — the system narrating its own reasoning.
3
Flash-of-content is a context failure When protected content briefly appears, users doubt whether the system is actually protecting their data. The loading state is not a UX nicety — it is a security layer.

Row Level Security

RLS: Enable → Deny All

RLS enabled + no policy = zero rows returned

Not a bug. RLS defaults to deny-all. The database would rather silently return zero rows than accidentally return the wrong rows.

RLS (definition): database-level rule that limits which rows each user can read or change.

Why it matters: it protects data even when app-layer logic has bugs.

Real-world scenario: payroll portal loads for all employees, but each person sees only their own salary row.

Toggle RLS on → reload app → nothing loads. That is intentional design. Next step is adding policy rules that open only the right rows.

24 / 37
Row Level Security

Read Policy (Who Can See What?)

Policy intent (plain language)

A signed-in user can read only rows where the row owner matches that signed-in user.

1
Identity comes from session token Database reads who the current user is from authentication context.
2
Ownership comes from row data Each row already stores owner ID.
3
Rows are filtered automatically Users see only rows where identity and ownership match.

Example: in a banking app, two users open the same "transactions" page URL, but each sees only their own history.

Defense in Depth

Without RLS vs. With RLS

Without RLS
A
Team relies on app-layer filtering Developers must remember to apply privacy filters in every query path.
B
One missed filter leaks data A single oversight can expose all user rows.
With RLS
A
Database enforces ownership by default Access rules travel with the table, not with each developer memory.
B
Bug impact is contained Even if app logic is imperfect, cross-user reads are still blocked.

Code-level filtering and database-level enforcement are not redundant — they are two different lines of defense. Your code will have bugs. RLS is the safety net.

Dedicated comparison example

Situation: hospital nurse opens patient list.

Risk without RLS: one missed filter could expose all patient charts.

Safer with RLS: database limits rows to assigned patients, even if app query is imperfect.

Class prompt: In healthcare, which is safer for patient privacy: relying only on UI checks or enforcing row-level rules in database?

Agentic AI

RLS Is Non-Negotiable for Agents

The Bug

Agent uses a high-privilege service key in user-facing flow.
Data request asks for all rows without ownership filtering.
Result: Returns ALL users' data.

The Fix

Agent acts with end-user identity context, not admin context.
Row-level rules compare session identity to row owner automatically.
Result: Only that user's data.

Overlay Plane 2 agent locations on your architecture diagram. Every agent endpoint needs the same auth + RLS stack as every human endpoint.

Row Level Security

Write Policy Design: Create, Edit, Delete

Outgoing check (read/update/delete)

"Can this user touch this existing row?"

Example: a student cannot delete another student's assignment reminder.

Incoming check (create/update)

"Is the new row state allowed before saving?"

Example: user cannot create a task and assign ownership to someone else.

Best practice: protect both directions of data flow. If you only check reads but not writes, attackers can still inject unsafe rows.

Security Verification

Testing Your RLS

After writing policies, run behavior tests (not just "it compiles").

Test script in plain English
1
Act as User A Open app using A's login context.
2
Try to access User B's row Expected result: zero visible rows, always.
3
Repeat for write actions Ensure create/update/delete rules also block cross-user actions.
Test every policy. A missing RLS policy means any authenticated user can read/modify any row. This is the #1 Supabase security mistake.
Test CaseExpected ResultPass Criteria
Same-user query Returns user's own rows count > 0
Cross-user query Returns zero rows count == 0 always
Unauthenticated query Request blocked 401 or empty result
Admin / service role Broadened access (intentional) Only via server-side, never client

Key metric: Cross-user queries must always return 0 rows.

PRIMM: Predict

Find the Vulnerability

Scenario A: Input Safety

Situation: note form accepts unlimited text and raw HTML.

Risk: malicious script can be stored and shown to other users.

Safer practice: validate length and sanitize before save.

Scenario B: Route Protection

Situation: login check exists only in browser UI.

Risk: direct API calls bypass UI redirect behavior.

Safer practice: enforce checks on server for every protected endpoint.

Scenario C: Data Access Rules

Situation: user-specific table has no row-level rule.

Risk: any authenticated account can query all rows.

Safer practice: add ownership-based row policy on all user data tables.

Class prompt: Rank A/B/C from "most likely in beginner projects" to "most damaging in production."

30 / 37
PRIMM: Investigate

Vulnerability Debrief

A
Missing input validation No length limit, no sanitization. An XSS payload can go straight into the database and render in another user’s browser. Fix: add clear limits and sanitize content on the server before writing.
B
Client-side-only auth check The auth check runs only in browser UI logic. JavaScript can be disabled. A direct API call bypasses it entirely. The redirect is a UX affordance, not a security control.
C
No RLS enabled The Supabase anon key is public — it’s in your client-side bundle. Without RLS, that key is a skeleton key to your entire table. This is the default state of a new Supabase project.

AI Security Checklist

These three are the top vulnerabilities in AI-generated full-stack code:

1
Missing input validation Does the form check length, type, and special characters on the server?
2
Client-side-only auth Is there a server-side auth check, or only a browser redirect?

No Row Level Security Is RLS enabled on every table with user-specific data?

Run this checklist every time AI generates a form, a protected route, or a database insert.

Spec-Driven AI

CLAUDE.md as Context

What this file should declare
1
Data model intent What entities exist and who owns them.
2
Security rules Which routes need auth, and which tables need row-level protection.
3
Validation boundaries What input limits are mandatory before data writes.

Spec-driven workflow (definition): define requirements before implementation so humans and AI follow the same source of truth.

Why it matters: consistent specs reduce rework, security omissions, and contradictory outputs.

Example: when teaching team project handoff, everyone reads one requirements file instead of relying on memory.

When the data model, auth requirements, and RLS rules are in CLAUDE.md, the AI generates correct Supabase queries on the first try. No prompt iteration, no back-and-forth.

Key insight: A chat prompt is ephemeral. A CLAUDE.md is re-read at the start of every session — the 10th query is as accurate as the 1st.

Example proposal (no command syntax)

Title: Add task filtering

Why: students need to quickly separate done vs pending work.

What changes: add filter control, update task list behavior, keep "show all" default.
