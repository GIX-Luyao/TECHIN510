Why This Order Matters

The 40 / 20 / 40 Discipline

40
Planning Understand the user, the constraints, the architecture — before any prompt that produces code.
20
Coding The shortest part of the loop. AI handles most of the typing once the spec is clear.
40
Testing Verification, regressions, edge cases. Equal weight to planning — not an afterthought.

“Vibe coding hangover” is the cost of skipping the first 40% and the last 40%.

Today's tools each defend one of those buckets so you can’t skip them by accident.

Tool 01

gstack

A senior team, summoned through slash commands.

gstack is an open-source pack of Claude Code skills authored by Garry Tan's team. Each command turns Claude into a specific kind of reviewer or operator — CEO, eng manager, designer, QA — with a sharply scoped prompt and checklist.

Install once. Available in every project. Today we use it for the planning stage.

Repository: github.com/garrytan/gstack

05 / 17
Tool 01 — In Practice

The Commands That Earn Their Keep

  • /office-hours
    YC-style forcing questions on demand reality. Surfaces the “who would actually use this?” you’ve been avoiding.
  • /plan-ceo-review
    Challenges premises. Asks whether you’re building the 1x version when a 10x version is in reach.
  • /plan-eng-review
    Locks in architecture, data flow, edge cases. Produces the spec you’ll execute against.
  • /plan-design-review
    Designer’s eye on the UX flow before pixels exist. Catches confusing states early.
  • /design-consultation
    Researches the visual landscape, proposes a complete direction. Useful starting point for the design stage.
  • /review  ·  /ship
    Pre-merge diff review and a guarded ship workflow. Keeps the “commit before testing” instinct in check.

Tool 02

Claude Design

A design surface that outputs real HTML, not screenshots.

Most AI-generated UI looks generic because the model is generating tokens of code from scratch every time. Claude Design flips that: it owns a dedicated visual workspace, lets you iterate on a design, and exports a real prototype file you can read, fetch from your IDE, and adapt.

Without it

Paste “design me a settings page” into chat → cardboard-cutout layout, generic palette, no consistency between screens.

With it

Iterate visually, hold a coherent system across screens, hand the export to Claude Code as a reference instead of a starting prompt.

08 / 17
Tool 02 — Workflow

How To Use It Well

1
Open the Design surface inside Claude It’s a separate workspace — not a chat reply. Treat it like Figma, not autocomplete.
2
Describe the screen and the intent “A settings page” is weak. “A settings page where the user can rotate API keys, with destructive actions visually separated” is real.
3
Iterate visually, then export Refine inside the surface. When it’s right, export the prototype HTML. You now have a versioned design artifact.
4
Hand the export to Claude Code as a reference Not as final code. Tell Claude: “match this layout and palette in our Next.js components.”

Tool 03

superpowers

A library of skills that make Claude work like a disciplined engineer.

Each “superpower” is a structured workflow that Claude Code activates automatically when it sees the right kind of task. They’re not chat templates — they enforce process. Today’s focus is the one that runs the build stage:

superpowers:test-driven-development

“Use when implementing any feature or bugfix, before writing implementation code.”

Companion skills: brainstorming, writing-plans, executing-plans, systematic-debugging, verification-before-completion.

11 / 17
Tool 03 — The Loop

Red · Green · Refactor · Verify

01 — Red
Write a failing test
superpowers refuses to write implementation code first. The test names the contract.
02 — Green
Smallest passing change
Just enough code to make the new test pass — no “while we’re here” refactors.
03 — Refactor
Clean while green
Tests still pass. Improve names, extract helpers, only after the bar is green.
04 — Verify
Don’t self-grade
superpowers:verification-before-completion runs the suite and inspects output before claiming done.

Don’t let AI mark its own homework. The verification skill exists because models will tell you they finished while the test runner is still red.

Tool 04

Playwright CLI

End-to-end browser tests, versioned alongside your code.

Unit tests prove your functions work. Playwright proves the app works — that a real browser, against your real Next.js + Supabase stack, can sign up, click through, and not blow up.

Manual QA

Click through the flow yourself before every push. Forget half the edge cases. Find the regression in production.

Playwright

Tests survive the week. Run on CI before deploy. Catch the regression you’d never click through manually.

13 / 17
Tool 04 — The Three Commands

You Only Need Three

  • npm init playwright@latest
    One-time setup. Adds playwright.config.ts, a sample test, and the browser binaries.
  • npx playwright codegen
    Opens a real browser. Click through your app — Playwright transcribes each interaction into a runnable test. The fastest way to write your first one.
  • npx playwright test ——ui
    Run the suite in interactive UI mode. Step through, see screenshots at each action, debug failures visually.

Lab task: for your Next.js project, write one Playwright test that covers the critical sign-up → first-action flow. That single test will catch 80% of the regressions you’d ship by accident.

What I’m Asking You To Do

This Week’s Recommendation

For your Week 5 Next.js + Supabase project, run the full stack at least once. Even if it feels like overkill for a small lab — that’s the point of practicing it small.

  • Run /office-hours and /plan-eng-review in gstack before writing any code
  • Generate one anchor screen with Claude Design and reference it from your components
  • Use superpowers:test-driven-development for the trickiest piece of business logic
  • Write one Playwright test that walks the critical user flow end-to-end
  • In your reflection, note which tool surprised you — positively or otherwise

Grading note: I care about whether you tried the workflow and learned something, not whether you used every tool perfectly. Reflection beats checklist completion.
